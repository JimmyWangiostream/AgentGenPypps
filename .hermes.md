## Core generation principle

Use this separation:

```text
TC decides WHAT to test.
LLM Wiki supplies domain/spec/generation-policy/model-default knowledge.
GitNexus or direct PyPPS code search supplies HOW PyPPS code should be written.
Decision layer resolves conflicts using source priority.
Generated artifacts must explain every fallback and unresolved detail.
```

Do not silently infer details. If a PyPPS API, enum, IDN, reset primitive, LUN rule, transfer length, expected value, or implementation method is not grounded, record it in `assumptions.json` and/or `validation.json`. Use `TODO_REVIEW` in the generated `.py` when code still needs human confirmation.

## Source priority rules

### Expected behavior / test intent

For decisions like expected values, pass/fail behavior, whether a flag should reset to 0, or what feature is being validated:

```text
1. TC explicit behavior
2. UserPrompt explicit override / clarification
3. CustomerReq project/customer behavior
4. Spec protocol baseline from LLM Wiki
5. ModelDefault fallback only if allowed
```

### Implementation details

For decisions like PyPPS APIs, class/function signatures, imports, loop count, LBA choice, transfer length, compare method, random seed, code style, reset primitive:

```text
1. TC explicit implementation detail
2. UserPrompt implementation rule
3. CustomerReq implementation-method
4. GitNexus / /home/weikai/Script existing implementation
5. Project convention
6. ModelDefault fallback
7. BLOCKER or TODO_REVIEW
```

Spec can validate protocol behavior, but PyPPS code must be grounded by `/home/weikai/Script` or GitNexus/code search.

## High-level pipeline

Follow this pipeline. The key rule is: **one TC produces one Pattern folder and one Pattern `.py`, but generation granularity is flow-level and stateful**.

```text
TC markdown
  -> parse full TC into global context + Flow IR + Step IR
  -> for each flow:
       preserve ordered steps inside the flow
       build data/state dependency graph
       run LLM Wiki retrieval per flow/action group
       run PyPPS code retrieval per flow/action group
       apply decision layer with global context + flow state + prior steps in same flow
       generate one stateful Flow Plan, not isolated step snippets
       validate flow coverage and dependency satisfaction
  -> assemble all Flow Plans into Pattern IR
  -> generate one .py
  -> generate flow-scoped retrieval.md / assumptions.json / validation.json
  -> run final syntax + structure + dependency validation
```

Do **not** generate the whole TC in one broad pass: retrieval becomes noisy and expected values can be mixed across flows. Also do **not** generate each step as an isolated code snippet: step-to-step variables such as `write_record`, `reset_type`, `expected`, pre-reset flag state, selected LUN, and previous read/write values can be lost. Use the middle path: **flow-scoped retrieval + flow-level stateful generation**.

## TC parsing expectations

Read the TC markdown first. Extract:

```text
- Pattern name / TC name
- Purpose / objective
- Preconditions
- Main flow phases
- Step IDs
- Action type
- Target flag/attribute/command
- Parameters: LUN, LBA range, transfer length, loop count, reset type, expected value
- Expected result
- Missing fields
```

Classify steps when possible:

```text
- TEST UNIT READY
- READ CAPACITY
- READ FLAG / SET FLAG / CLEAR FLAG / TOGGLE FLAG
- READ ATTRIBUTE / WRITE ATTRIBUTE
- WRITE(10)
- READ(10) / compare
- reset / POR / HW_RESET / ENDPOINT_RESET / UNIPRO_RESET / LINKSTARTUP
- START STOP UNIT / power mode transition
- setup / cleanup
- loop / burn-in / randomized coverage
```

## Flow-scoped stateful generation

Use this as the required granularity:

```text
One TC = one Pattern folder + one Pattern .py
One flow/phase = one generation unit / usually one step<number>() method
One flow contains ordered steps + shared flow state + data dependencies
Step-level retrieval is allowed; isolated step-level code generation is not allowed
```

Before generating code for a flow, build a state/dependency graph. Capture:

```text
- values produced by earlier steps: old flag value, attribute value, write_record, selected LUN, reset_type, expected value
- values consumed by later steps: compare targets, post-reset checks, restore values, read/write LBA and length
- global state shared across flows: seed, test_lun, write_record, capacity, transfer length policy
- flow isolation/cleanup requirements: flags to set/clear before next flow, reset effects, post_process cleanup
```

A flow plan should explicitly list producers and consumers, for example:

```text
Step 1.2 random_write -> produces write_record
Step 1.3 read_compare -> consumes write_record
Step 1.4 reset loop -> produces reset_type
Step 1.5 read_flag expected -> consumes reset_type + pre-reset flag state
```

Generate the whole ordered flow method at once so dependent variables remain coupled. For example, do not generate `_do_reset(reset_type)` separately from the assertion that consumes `reset_type`; keep the reset primitive and expected-value decision in the same flow plan.

Represent optional/conditional flows, such as FFU during burn-in, explicitly. If required implementation details are missing (firmware path, target version, trigger policy, transfer length), generate a disabled `TODO_REVIEW` placeholder or blocker rather than silently dropping the flow.

## LLM Wiki retrieval

Use the persistent embedding index. Example query:

```bash
CUDA_VISIBLE_DEVICES='' python3 -m auto_pattern_gen.llm_wiki.retriever query \
  "<TC name + key step text + normalized keywords>" \
  --chunks .pattern_kb/index/chunks.jsonl \
  --embedding-index .pattern_kb/index/embeddings/sentence-transformers--all-MiniLM-L6-v2 \
  --top-k 10 \
  --json
```

The retriever returns:

```text
BM25 score
Dense score
RRF score
chunk_id
page_path
title
source_type
priority
authority
text_preview
```

Always preserve BM25, Dense, and RRF scores in `<Pattern>.retrieval.md`. Do not only list the chunk names.

If the embedding index is stale or missing, rebuild it:

```bash
CUDA_VISIBLE_DEVICES='' python3 -m auto_pattern_gen.llm_wiki.retriever build-embeddings \
  --chunks .pattern_kb/index/chunks.jsonl \
  --dense-backend sentence-transformer \
  --embedding-model sentence-transformers/all-MiniLM-L6-v2 \
  --out .pattern_kb/index/embeddings/sentence-transformers--all-MiniLM-L6-v2
```

Expected dense backend for generated reports:

```text
persistent:sentence-transformer
```

## Pattern `.py` structure

Prefer this structure unless current PyPPS references indicate otherwise:

```python
import random

import package_root
from Script import api
from Script.api import shared
from Script.api import cmd_seq as ExecuteCMD
from Script.pattern.pattern_template import UFSTC
from Script.pattern.pattern_logger import logger


class Pattern(UFSTC):
    def pre_process(self) -> None:
        ...

    def is_support(self) -> bool:
        return True

    def step1(self) -> None:
        ...

    def step2(self) -> None:
        ...

    def post_process(self) -> None:
        ...


run = Pattern().run
if __name__ == "__main__":
    run()
```

`UFSTC.process()` automatically runs methods named `step<number>` in numeric order.

Use trace comments sparingly in the `.py`; put detailed trace in `retrieval.md`.

When a generated line depends on an unresolved mapping, write a clear comment like:

```python
# TODO_REVIEW: TC says LINKSTARTUP Reset, but current PyPPS reset enum has no explicit LINKSTARTUP_RESET.
```

## Assumptions JSON format

`<Pattern>.assumptions.json` should record all fallbacks and unresolved items:

```json
{
  "pattern_name": "...",
  "generated_at": "...",
  "mode": "auto_accept_for_HARD_DEFAULT_and_recorded_SOFT_ASSUMPTION",
  "assumptions": [
    {
      "id": "A001",
      "scope": "flow_1",
      "category": "SOFT_ASSUMPTION",
      "field": "compare_method",
      "value": "api.CompareMethod.HW_COMPARE",
      "source": "llm_wiki/model-default/auto/default-compare-method.md#chunk-002",
      "depends_on": ["flow_1.write_record", "flow_1.READ_COMPARE"],
      "reason": "TC requires Data Match but does not specify HW vs SW compare."
    }
  ],
  "blockers": [],
  "todo_review_count": 0
}
```

Categories:

```text
HARD_DEFAULT
SOFT_ASSUMPTION
TODO_REVIEW
BLOCKER
```

Use `BLOCKER` if the core test intent or expected result is unknown and cannot be safely inferred. Include `scope` (`global`, `flow_<n>`, or `flow_x`) and `depends_on` for every assumption/TODO_REVIEW so reviewers can see which ordered flow and state dependency is affected.

## Validation steps

After writing the Pattern:

```bash
python3 -m py_compile generated/<PatternName>/<PatternNamePythonSafe>.py
```

Also parse with Python `ast` or inspect the file to verify:

```text
- class Pattern exists
- pre_process exists
- one or more step<number> methods exist
- post_process exists
- run = Pattern().run exists
- TODO_REVIEW count is reflected in validation.json
- every flow has a `flow_coverage` entry
- every variable read by a flow has a producer in global context or an earlier ordered step in the same flow
- reset_type and expected-value assertions are generated in the same flow plan
- optional flows are represented as implemented, disabled TODO_REVIEW, or blocker; they are not silently dropped
```

Remove transient `__pycache__` from generated output folders after validation so the delivery folder only contains the expected artifacts.

## ModelDefault policy

ModelDefault may be used only when higher-priority sources do not specify a detail. Every ModelDefault use must be recorded in `assumptions.json`.

Common allowed fallbacks:

```text
- deterministic random seed or runtime seed policy
- default compare method
- default generated file style
- reset coverage strategy when TC lists reset candidates but not scheduling
```

Do not use ModelDefault to fabricate:

```text
- core expected behavior
- unknown expected value
- missing flag/attribute IDN when no reliable source exists
- nonexistent PyPPS API
- unsupported reset primitive
```

For those, emit `TODO_REVIEW` or `BLOCKER`.

## PF010_0310 lessons learned

During PF010_0310-Normalize generation these unresolved items were found and must not be silently hidden in future generation:

```text
1. dLUNumWriteBoosterBufferAllocUnits API/IDN mapping
   TC says attribute IDN is 0x17, but current PyPPS AttributeIDN enum maps 0x17 to REF_CLK_GATING_WAIT_TIME and has no exact dLUNumWriteBoosterBufferAllocUnits name.
   Do not write this attribute unless a newer grounded source confirms the correct API/IDN.

2. fWriteBoosterSupport enum missing
   TC appendix says fWriteBoosterSupport is 0x15, but current PyPPS FlagIDN enum has no matching name.
   If using numeric 0x15, record TODO_REVIEW.

3. LINKSTARTUP reset mapping
   TC says LINKSTARTUP Reset, but current PyPPS reset enum exposes HW_RESET, ENDPOINT_RESET, and UNIPRO_RESET.
   Mapping LINKSTARTUP to UNIPRO_RESET is a TODO_REVIEW unless a better grounded reference is found.
```
